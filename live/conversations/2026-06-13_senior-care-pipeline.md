## The product design (the iterated path the next seat needs)
PRODUCT = a per-account, isolated, snapshot-holding, SMS-native emergency-coverage matcher that resolves availability BY ASKING, NOT STORING. Code-first; AI only where code can't satisfy. AI lives UPSTREAM (the worker pipeline builds it; drift-repair heals it), and LATER in a year-two insight layer — never in the live coverage decision (human-in-the-loop). "Boring reliable tool, intelligent factory behind it."

AVAILABILITY DESIGN (iterated, operator-authored refinement): two grounded layers, both code.
- FOUNDATION SNAPSHOT: scheduled availability-poll texts at known times daily. Reply within ~20-min window = available; SILENCE = definitively unavailable (reason irrelevant). The known-time convention turns non-reply into SIGNAL, not missing data (evidence-of-absence). Labor pool maintains the dataset itself, near-zero cost, no stored-flag staleness. Caregivers can also cancel a regular shift at a preset days-in-advance rule, self-mark available to pick up, or poll-text the community on an open shift.
- RUNNING SNAPSHOT (emergency matcher): sits ON TOP of the foundation layer. On trigger (caregiver out): filter ClearCare schedule for not-booked (code) → rank by stored preferences (code) → text top N "shift open, reply YES to claim" → availability resolves by who replies (query reality at moment of need). Human owner makes the actual call.
WHY NO AI NEEDED: search-and-notify, not inference. SMS-native because a stressed coordinator at 6am wants to text, not log in (OpenClaw interface-reach lesson).

DATA ACCESS: ClearCare/WellSky HAS surfaces (corrects the earlier "no API" assumption): Connect API (FHIR REST, base connect.clearcareonline.com/v1/, /practitioners/ + /appointment/, JSON) + Insights Agreement (Third-Party API, JDBC). HARD CONSTRAINT: agreements forbid reverse-engineering/scraping; WellSky is HITRUST/HIPAA. So the cooperative-layer-from-outside must use the SANCTIONED surface (like building on Claude Code's CLI/MCP, not decompiling it). Forbidden: bot-driven UI scraping/mouse-mimic (grey zone + most fragile = breaks on every layout change, ironically the drift problem on the critical path). SANCTIONED + LOW-FRICTION: a licensed human (Katie or coordinator) triggers an export; tool automates everything after. BEST CASE to ask ClearCare: can it auto-EMAIL a scheduled report (daily CSV) to an address — zero ongoing human action. FLOOR (always works, no license): Katie exports her own data → tool ingests. DESIGN SEAM: abstract the data source behind a swappable ingest adapter NOW, so API-vs-export becomes a swap, not a rebuild. GENERALIZATION (product seed): build against the STANDARD (FHIR), not the vendor — same helper layer works with any FHIR system.

## The four-stage pipeline (stitched this session → spec 6c15b21d)
intake → problem-definition → decomposition → solution proposal. Each = a Tetraform session writing provenance-tagged output (confirmed/assumed/open) to a per-project mini-corpus; final stage composes from the whole chain. Mini-corpus = pull-based handoff (not fragile push-injection). Two-axis isolation: facts only from the project's own corpus (strict, all stages); methodology may be read from main corpus but ONLY in stages 3–4, read-only, never written back (main is write-sealed against client sessions always). Convergence: re-run only with NEW info (resolved unknowns); stop when remaining `open` rows don't change the proposal. The proposal SHOWS provenance to the client — trust spine made legible. Only stage 1 (intake) is LIVE; stages 2–4 are the next build.
