# SPEC — The Coordinator (deterministic mailbox-driving code loop)

*Status: BUILD SPEC (not deployed). Authored by control 2026-06-30, dispatched to a worker for build + peer review. The coordinator is the SHEPHERD'S loop shape (live/shepherd.py — the proven start/watch/verify/pace/repeat conductor) with the session-driver half swapped for a MAILBOX-driver half. Grounds: live/specs/gated_session_substrate.md (the corrected three-role design — Coordinator = a deterministic code loop in the shepherd's lineage that JUST coordinates; NOT a Tetraform session; no Oracle/deep-review bolted on); live/shepherd.py (the loop skeleton + halt discipline this reuses); live/box/seat_mailbox.py (the FIFO atomic-claim mailbox it drives); the no-self-sign-off / two-party deploy invariant. Do not deploy from this file; it specifies the build for operator sign-off.*

## 1. WHAT IT IS / IS NOT
The coordinator is a deterministic Python loop that drains the live mailbox and advances each work item through its review chain in CODE — claim the oldest queued item, determine the chain step it needs, run the no-self-sign-off checks deterministically, fire the courier op that advances it, log, loop. It JUST coordinates. It does NOT make contestable judgments, does NOT call any Oracle, does NOT run adversarial review — those were stripped from the design as over-build. Where a genuine judgment fork appears (rare), it surfaces the item to the operator/control rather than deciding; it never fabricates a decision to keep moving.

It is the shepherd's sibling: the shepherd drives ENGINE SESSIONS (agent/start -> watch /diag/engine -> verify write_receipt -> pace -> next, from a static battery.json). The coordinator drives the MAILBOX (claim oldest queued -> advance the chain -> fire courier op -> log -> next, from the live dynamic queue). Same loop shape; different unit of work.

## 2. REUSE FROM shepherd.py (read it; copy the proven structure)
- The loop skeleton: `http()` helper, JSONL event log (every action logged), `while` bounded by a budget, `time.sleep` pacing between iterations (rate weather).
- The honest-halt discipline: stop the loop on a condition it cannot safely handle rather than barrel on — the shepherd halts on provider-failure weather, missing receipt, operator modal. The coordinator halts on: an unparseable/ambiguous chain state, a courier op failing repeatedly, or a judgment fork that needs the operator. Halt + log + surface, never guess.
- DO NOT COPY the shepherd's line-13 hardcoded DIAG_KEY plaintext fallback. That is a live-credential exposure. The coordinator reads its key from a file (perms 600) or the vault, NEVER a literal in source. (Flag: the shepherd itself should be scrubbed of that literal — separate item, note it in your proposal.)

## 3. THE LOOP (one iteration)
1. CLAIM: claim the oldest queued actionable item (mailbox_fetch default = ORDER BY created_at ASC, atomic BEGIN IMMEDIATE — verify against live seat_mailbox.py). "Actionable" = a chain item the coordinator advances (a proposal awaiting routing, a signoff awaiting a commit, a result awaiting ack/close). It does NOT claim items meant for worker building (task/proposal addressed to any_worker for build) — scope by kind/to_seat so the coordinator and the worker pool do not fight over the same items. Builder: determine the exact actionable set against live mailbox semantics and STATE it.
2. CLASSIFY: determine the chain step deterministically from the item's kind + the chain state (e.g. a kind=signoff whose author differs from the bytes' author -> the item is ready to commit; a kind=proposal with no signoff yet -> route to a reviewer that is NOT the author).
3. CHECK (no-self-sign-off, in code): before advancing, verify the two-party invariant deterministically — the signer/reviewer seat MUST differ from the author seat; a "ready to land" item MUST cite a real kind=signoff row whose author_seat differs from the block author (the review-misclassification guard: a reject must never be read as a signoff). If the invariant is not satisfied, do not advance; route the missing step instead.
4. ACT: fire exactly the courier op that advances the item (mailbox_send to route a review to a distinct seat; commit_file/commit on a clean signoff; mailbox_ack/close on a spent result). One op per item per iteration; log the op + its result.
5. LOG + LOOP: append a JSONL event (item id, classified step, op fired, result), pace, repeat. On the halt conditions in §2, stop + surface.

## 4. WHAT IT EXPLICITLY DOES NOT DO
- It does NOT deploy to a watched path or install box code on its own — a STATE-CHANGING deploy (a box install, a watched-path commit that ships behavior) remains OPERATOR-GATED. The coordinator may fire the courier ops that ROUTE and that COMMIT already-signed-off artifacts per the two-party chain, but the human deploy authority is unchanged. Builder: draw this line explicitly — which ops the coordinator may fire autonomously (routing, acking, committing-a-signed-off-spec) vs which require the operator gate (box install, restart, deploy). When in doubt, surface, don't fire.
- It does NOT choose WHICH task a worker builds next — the FIFO queue does that. The coordinator routes review/signoff/commit chain steps, not build-task selection.
- It does NOT invent a seat. It routes to real seats (the pool any_worker, or a specific existing seat when that seat is the correct chain participant). Until the seat-registry/identity primitive lands, it relies on self-asserted seat fields like the rest of the system and is HONEST about that limit (same dependency the bootstrap gate notes).

## 5. WHERE IT LIVES
A sandbox/box-resident Python loop (form like shepherd.py), runnable as a process. NOT an engine Tetraform session (that would be the awkward squeeze the design rejected — routing is deterministic, not adversarial-grounding work). It reads its credential from a file/vault, drives the MAIN engine's courier, logs to JSONL (and optionally to operations_ledger if a courier surface is added later — out of scope for v1). It does NOT consume a Tetraform driver slot (it is a code loop, not an engine session) so it does not collide with farm sessions.

## 6. ACCEPTANCE (both directions)
- Drive a real chain end to end with NO operator routing: stage a proposal from seat X -> the coordinator routes review to a DISTINCT seat -> on a returned signoff (author != signer) the coordinator commits the artifact -> log shows each step + op. Demonstrate it advanced a chain the way control did by hand this week, hands-free.
- Catch the guard cases: (a) a proposal whose only "signoff" is actually a reject row -> the coordinator must NOT commit (review-misclassification guard); (b) a signoff whose author == the bytes' author -> the coordinator must NOT commit (no-self-sign-off); (c) a judgment fork / ambiguous state -> the coordinator HALTS and surfaces, does not guess. A coordinator that only drives the happy path is not done.

## 7. PROCESS
Worker-proposable. Author builds the loop + this spec's logic; a DIFFERENT seat reviews and signs off (no self-sign-off); operator gates the run + any state-changing op the coordinator would fire. Stage + propose; do not stand the loop running against the live mailbox until reviewed and operator-authorized — a loose coordinator firing courier ops on the live queue is exactly the blast radius review exists to contain. Read live/shepherd.py, live/specs/gated_session_substrate.md, and live seat_mailbox.py before writing; verify every mailbox-semantics claim against source, do not assume.
